How a follow-up gets queued

Inside onSend

Relevant area:

  • apps/web/src/components/ChatView.tsx:3150

If the user presses send while a turn is already running:

  • ChatView does not start a new turn immediately
  • it packages the current composer content into a QueuedFollowUpDraft
  • clears the composer
  • shows a success toast like:
    • “Queued follow-up”
    • or “Queued follow-up (N)”

So the “queue” behavior is handled directly in onSend.

How queued follow-ups are actually sent

sendQueuedFollowUpDraft(...)

Relevant area:

  • apps/web/src/components/ChatView.tsx:3616

This function sends the queued draft by dispatching:

  • api.orchestration.dispatchCommand({ type: "thread.turn.start", ... })

It also:

  • creates optimistic user message UI
  • uploads images as data URLs
  • persists thread settings for next turn
  • marks local send/dispatch refs
  • restores the draft on failure

Auto-dispatch effect

Relevant area:

  • apps/web/src/components/ChatView.tsx:3753

A useEffect watches queue + runtime conditions and automatically sends the first queued draft when all blockers are clear.

It will not send if any of these are true:

  • current turn still running
  • send already busy
  • connecting
  • dispatch already in flight
  • pending approval exists
  • pending user input exists

When eligible:

  • it takes queuedFollowUps[0]
  • sends it
  • removes it from queue on success
  • currently also shifts it out on failure after restoring it to composer state

So queued follow-ups are strict FIFO by default, with a manual “send next” reorder control.

Steer-now vs queue-follow-up

This distinction is important.

sendStreamingComposerDraft(...)

Relevant area:

  • apps/web/src/components/ChatView.tsx:3451

This is used for Steer now while a turn is running.

It sends through:

  • api.orchestration.sendStreamingMessage(...)

with:

  • streamingBehavior: "steer" or "followUp"

In the current ChatView wiring:

  • the Steer now button calls sendStreamingComposerDraft("steer")
    • apps/web/src/components/ChatView.tsx:5185

By contrast:

  • normal submit during a running turn uses the local queue path from onSend

So behavior splits like this:

  • Steer now = inject follow-up into the active running turn via streaming API
  • Queue follow-up = hold a draft locally and send a new turn automatically after the current turn settles
